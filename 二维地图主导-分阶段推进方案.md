# 二维地图主导、全景为辅：分阶段推进方案

> 供组内讨论用。目标：用户点击后进入**多层二维地图**，通过**搜索**选定起点/终点，在二维地图上显示**推荐路径**；全景图仅作为用户**可选**的辅助展示。当前全景原型保留，但不再作为主入口。

---

## 0. 目标与原则（先对齐）

- **主路径**：用户进入 → 选楼层 → 在平面图上看点线 → 搜索/点选起点终点 → 显示推荐路径（基于现有 Dijkstra + `graph.json`）。
- **辅路径**：从某个节点（或路径某一段）「进入全景」浏览；返回地图时不丢失起终点状态。
- **数据**：每层一张底图 + 一层一份（或一份里按 `floor` 分）的 `nodes`（含 **`x_px`/`y_px` 或归一化坐标**）+ `edges` + `facilities`；跨层用「楼梯/电梯」节点或显式 **层间边** 连接。

---

## 阶段 1：单层二维地图 MVP（不接全景）

1. **定坐标系**：选定一层平面图 PNG/JPG，约定 `meta.image`、`meta.width/height` 或节点上 `x_px, y_px`（与像素一致最易上手）。
2. **做一个纯前端的「地图页」**（可先单独 `map.html`，与 `panorama.html` 并列）：
   - 底图 `<img>` + 上层 `<svg>` 或 `<canvas>` 画节点小圆点、边（淡线）、路径高亮（粗线）。
3. **起终点交互**：先不做搜索——**点击节点**或下拉选 `node.id` 作为起点/终点，调用 **浏览器内 Dijkstra**（把 `indoor_nav` 的逻辑抄一份到 `.js`，或先写极简内联）或请求本地 Python 小 API（二选一，见阶段 4 说明）。
4. **验收**：单层上路径与 JSON 一致；缩放/窗口变化时坐标仍对齐（若用像素坐标，注意 `object-fit` 与容器缩放时用同一比例换算）。

---

## 阶段 2：多层（楼层切换 + 跨层连通）

1. **数据拆分**：`floors/1/graph.json` + `floors/1/plan.png`，或单文件里 `nodes[].floor` + 多 `meta.images[floor]`。
2. **UI**：楼层 Tab / 侧边列表；切换楼层时换底图与当前层子图；**跨层边**只在「楼梯口/电梯」类节点上连接（同一 `id` 跨层或 `stair_1f` ↔ `stair_2f` 两条记录 + 固定权重如「爬楼时间」）。
3. **路径展示**：若最短路径跨层，在 UI 上提示「经北侧楼梯至 2 层」并在两层各画一段折线（或一层画完切层自动高亮下一层段）。
4. **验收**：跨层最短路正确、不会在未连通层间「穿墙」。

---

## 阶段 3：搜索定起终点（主交互）

1. **检索数据源**：用 `nodes[].label`（及可选别名数组 `aliases: []`）建一个前端索引；中文可先做 **子串匹配**，后续再加拼音首字母若需要。
2. **UI**：两个搜索框（起点 / 终点）+ 下拉联想；选定后地图上高亮对应节点。
3. **设施**：设施若挂在节点上，搜索可同时搜 `facilities[].label` 与类型，解析到 `nodeId` 作为终点。
4. **验收**：不点地图仅靠搜索也能完成「从 A 到 B」和「从当前位置到洗手间」类流程。

---

## 阶段 4：与现有 Python 服务整合（选一条主线）

- **方案 A（推荐起步）**：地图页仍静态，**算路全在浏览器 JS**（与 `node_nav` 同构），Python 只继续提供静态文件服务或以后打包 exe——部署最简单。
- **方案 B**：在 `server_main.py` 同进程加 `GET /api/route?from=&to=` 或 `POST`，内部 `import indoor_nav`，返回 `path_node_ids` + `labels`；前端只负责画线。适合以后要做 **服务端校验、多客户端、日志** 时再加重。

**建议**：先定一条，避免两条都算路、两套数据不同步。

---

## 阶段 5：全景降为「可选辅助」

1. **入口调整**：首页或 `map.html` 为主入口；`panorama.html` 从地图某节点「查看实景」进入，URL 带 `sceneId` 或 `nodeId → sceneId` 映射表。
2. **映射表**：在 JSON 里为节点增加可选字段 `panoramaSceneId`（无则按钮灰掉）。
3. **验收**：用户全程可不打开全景仍能完成导航；打开全景仅为增强体验。

---

## 阶段 6：体验与交付

1. **Legend**：图例（起点/终点/路径/不可达提示）。
2. **错误处理**：起终点未选全、不连通、JSON 缺字段时的提示。
3. **打包**：exe 仍起本地服务，**默认打开 `map.html`** 而非 `panorama.html`；全景链接仍可用。

---

## 依赖关系小结（排期参考）

```text
阶段1 单层地图 + 坐标 + 点击选点 + 画路径
   ↓
阶段2 多层 + 跨层边 + 楼层切换
   ↓
阶段3 搜索起终点（+ 设施）
   ↓
阶段4 定算路在浏览器或 Python API（二选一锁死）
   ↓
阶段5 全景改为可选入口 + node→scene 映射
   ↓
阶段6 打磨与默认首页调整
```

---

## 风险与注意点

- **坐标与底图**必须同一套比例；换图要批量更新坐标或改用归一化坐标。
- **跨层图**若每层单独 JSON，注意楼梯节点 **id 成对** 与边权一致，避免重复计算。
- **性能**：室内节点数小，Dijkstra 在前端足够；不必为上导航单独引入重型框架。
