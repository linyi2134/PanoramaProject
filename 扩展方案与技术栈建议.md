## 建议的下一步（按性价比排序）

### 1. 全景配置数据外置化

对齐文档中的 `panoramas.json` 思路：将热点从硬编码的 `panorama.html` 中拆出，改为由 `panoramas.json`（或合并进 `data/*.json`）加载。便于增删场景、修改文案与角度，并为后续「设施节点」与全景节点对齐打基础。

### 2. 路网与设施数据 + 最短路径

按文档附录中的 `nodes` / `edges` / `facilities` 结构准备 JSON；算法使用 Dijkstra 即可（可参考 indoor-wayfinder 的数据结构与思路）。

- **纯前端**：用 JavaScript 读取 JSON，在浏览器内完成路径计算并展示文字指引。
- **后端集中**：在现有 Python 服务中增加极简 API，返回路径结果（仍不必引入 Qt）。

### 3. 「当前位置」与路径结果的 UI

通过下拉框或列表选择当前逻辑节点（节点名即可，不必依赖 GPS），选择设施类型后显示最近设施与路径。简易 2D 示意可用 SVG/Canvas 绘制折线，或使用楼层静态示意图 + 高亮节点。

### 4. 用户标注（方案第 3 点）

浏览器内可先用 `localStorage` 实现「仅本机、仅本人」的私有标注。若需要导出、多端同步或规范存档，可由 Python 服务提供读写 JSON 的接口。文档中的 SQLite 为可选项，课程阶段使用 JSON 通常足够。

### 5. 发布形态

已有 `build_exe.py` 思路时，可继续采用 **PyInstaller 打包小型服务器 + 静态资源**，或双击启动本地服务，仍能体现「桌面交付」感，无需先学习 Qt。

## 是否采用 Qt？是否换技术栈？

**不一定需要 Qt**，理由如下：

- 方案中选择 Qt 主要是为了 **窗口 + OpenGL 球面渲染**；当前已用 **Web 全景引擎（Pannellum）** 替代，规避了 OpenGL/着色器调试成本，属于合理取舍。
- **设施查找、图算法、JSON 数据** 与具体 UI 框架无关，用 Web 或 Python 均可完成，且在数据结构与算法层面与参考项目 indoor-wayfinder（React/TypeScript）更易对齐。
- 团队不熟悉 Qt 时，强行切换往往将时间消耗在框架学习上，而非课程希望展示的「导航与数据」能力。

**可考虑 Qt（或 C++/Qt）的典型情况**：

- 课程 **明确要求** 桌面程序、C++/Qt 或指定技术栈；
- 必须 **离线且不依赖系统浏览器**，或对安装包形态有硬性要求；
- 需要 **重度自定义 3D/全景渲染**（超出 Pannellum 能力范围）。

**若希望桌面壳但不想投入 Qt 学习成本**，可考虑：**Tauri / Electron** 包裹现有网页，或继续 **Python + 本地 HTTP + 浏览器**（当前方案演进成本最低）。

## 结论摘要

- **优先推进**：路网与设施 JSON、Dijkstra、与全景场景 ID 对齐；全景配置外置；再完成标注与路径展示的 UI。
- **技术栈**：在课程未强制 Qt 的前提下，**延续「Python 本地服务 + Web 前端」** 通常能最快补全方案中的功能；**仅当出现上述硬性约束时** 再投入 Qt，否则性价比偏低。
